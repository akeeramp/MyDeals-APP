My Deals Code Quality Standards

This is the basic standards for My Deals Code Quality in the User Interface and Middle Tier.

Check In Process

Expectations
  • All checked-in code should compile as to not impact other developers
  • All development should be done with local code pointed to the Dev DB Server (Standard Visual Studio practice)
Scope
  • Check-ins should be scoped to a single defect or user story.
  • GitLab check-in should have comments on the check-in with the User Story or Defect number in the comment
  • Developers work in a single GitLab working location. Upon selected release stages, the working version will be branched, however all new development will be done in the GitLab folder path: MyDeals - GitLab
Lightweight
  • Keep LOC (Lines of Code) minimal. More code reviews are better than large code reviews
  • If possible (code must work on check-in), check in large User Stories at convenient milestones to prevent large code reviews and more code redesign post the reviews.
  • If code does not compile or code is NOT ready to be promoted through the environments, "git stash" the changes in GitLab until ready to formally check-in.

Code Review Request

Type of Code Review
  • Small LOC check-ins (typically < 200 LOC) can rely on GitLab Code Review
  • Large check-ins (User Stories typically > 200 LOC) need a Code Review Meeting prior to checking in code
Type of Code Review
  • Code must be checked in prior to making a code review request.
  • Every code check-in MUST be reviewed by the UI/MT Tech Lead (Michael Tipping and Saurav Kundu)
  • UI/MT Tech Lead will monitor check-ins for missing code reviews to catch check-ins without a requested code review. This will be done through automated emails from GitLab upon developer code check-in. Lead will manually match email against code review request to locate checked code without a code review.
